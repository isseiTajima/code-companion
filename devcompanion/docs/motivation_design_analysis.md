# サクラのモチベーション向上設計分析

作成: 2026-04-08

---

## 1. 現状の問題構造

### 1-1. 設計の根本的なズレ

サクラの当初ゴールは「ユーザーの行動を観察してコメントする」だった。しかし実際には：

- LLMは本質的に画面を「見ていない」（FSSensor が変化を検知するだけ）
- 「何をしているか」の具体的情報をLLMに渡すと、技術を知らないサクラが頓珍漢なコメントをする
- 結果、曖昧な観察コメントが増え、ユーザーには「的外れ」か「無意味」に聞こえる

→ **観察コメントを完全に捨て、情感的な同伴（emotional companionship）に絞るべき**

### 1-2. 現在のアーキテクチャ上の制約

```
MonitorEvent → poolCategory（6種類） → Pool → 汎用セリフを取り出す
```

**問題:**
- プールはカテゴリ単位で事前生成される。生成時に「今のセッション文脈」がほとんど入らない
- `SituationHint`（セッション時間・コミット数）は最近追加されたが、バッチ生成時の軽いヒントに過ぎない
- 同じカテゴリ（working）が大半を占めるため、「いまこの瞬間」感がなく汎用的に聞こえる

### 1-3. whack-a-mole バリデーションの限界

現状は「悪いフレーズを禁止する」事後処理で品質を保っている：

```
「絶対いけます」→ BAN → LLMが「のが素敵ですね」を使い始める → BAN → ...
```

- LLMは禁止されるたびに別の「型にはまったフレーズ」を使う
- 根本は「モチベーションを上げる発言」の軸がないことで、LLMが汎用的励ましに走る
- フレーズを禁止するより、**何を言うべきかのポジティブな制約**を強化する方が効果的

## 2. モチベーション設計の方向性

### コアコンセプト: 「セッションの感情的文脈に乗る」

ユーザーのモチベーションが上がる瞬間は次の3パターン：

| パターン | 状況例 | 欲しい言葉 |
|---------|--------|-----------|
| **勢い** | 調子いい、リズムに乗っている | 「一緒に乗ってる感」の共鳴 |
| **突破** | 詰まってたのが解決した | 「ちゃんと見てた」という承認 |
| **持続** | 長時間地味に続けている | 「すごいよ」という静かな賞賛 |

**現状のworking/achievement/struggleカテゴリはこの3つをひとまとめにしている**。
これを分けるだけで発言の精度が大幅に上がる。

---

## 3. 具体的な設計改善案

### 案A: セッションアーク追跡（インパクト大・コスト中）

**概念:**
セッション内の「感情的な弧（arc）」を追跡し、今どのフェーズにあるかをセリフ選択に使う。

```
フェーズ検出ロジック:
  - FRESH_START:  セッション開始 0〜15分、まだコミット0
  - MOMENTUM:     ActiveEdit が続いていて、failStreak == 0
  - GRIND:        30分以上経過、明確な成功もないが動き続けている
  - BREAKTHROUGH: 直近 10〜60分以内に struggle→success の遷移があった
  - DEEP_WORK:    StateDeepWork が継続、外部刺激への反応なし
```

**実装イメージ（SpeechGeneratorに追加）:**
```go
type SessionArc int
const (
    ArcFreshStart SessionArc = iota
    ArcMomentum
    ArcGrind
    ArcBreakthrough
    ArcDeepWork
)

func (sg *SpeechGenerator) currentArc(now time.Time) SessionArc {
    sessionMins := int(now.Sub(sg.codingSessionStart).Minutes())
    timeSinceLastSuccess := now.Sub(sg.lastSuccessAt)

    if sg.lastSuccessAt.After(sg.lastFailAt) && timeSinceLastSuccess < 30*time.Minute {
        return ArcBreakthrough
    }
    if sg.failStreak >= 2 && sessionMins > 20 {
        return ArcGrind
    }
    if sg.successStreak >= 2 {
        return ArcMomentum
    }
    if sessionMins < 15 {
        return ArcFreshStart
    }
    return ArcGrind
}
```

**効果:** 同じ「working」イベントでも ArcBreakthrough なら「さっきのやつ、解決できましたね」系、ArcGrind なら「ずっとやってますね、すごいです」系と使い分けられる。

### 案B: プールカテゴリの感情的分割（インパクト大・コスト中）

**現状のカテゴリ問題:**
```
working:     汎用すぎる（ほぼ毎回使われる）
achievement: GitCommit/Successの後にしか出ない
struggle:    LongInactivity専用になっている
```

**提案: 感情軸での再分類**

```
momentum    ← 「今いいリズムですね」「乗ってきてます」系（現working + 勢い系achievement）
endurance   ← 「ずっとやってますね」「こっちまで根気もらいます」系（長時間grind専用）
breakthrough← 「それですよ！」「さっきのやつ、通りましたね」系（success後/commit後の勢い系）
companion   ← 「何があっても隣にいます」系（idle/heartbeat、ただし観察ゼロ）
```

**実装コスト:** poolCategory() の返り値ロジックの変更 + ja.yaml/en.yaml への examples 追加

### 案C: 発言タイミングの感情的最適化（インパクト中・コスト低）

**現状の問題:** FrequencyController は「何分経ったか」でしか制御していない。

**改善:** 「感情的なタイミング」で優先的に発言する。

```
優先発言タイミングの追加:
1. failStreak が 3 以上になった直後（詰まってる検知）
2. failStreak → 0 遷移（解決直後）
3. セッション30分/60分/90分の区切り
4. 深夜（23時以降）に作業が続いている
```

上記タイミングでは isRoutine クールダウンを一時的に短縮（または無視）して発言機会を作る。

### 案D: SituationHint の強化（インパクト中・コスト低）

**現状:** `buildSituationHint(sessionMins, todayCommits)` は時間とコミット数のみ。

**追加できる情報（技術知識不要・観察可能）:**

```go
func buildSituationHint(sg *SpeechGenerator, now time.Time) string {
    // 現在の案（既存）
    // + 以下を追加:

    // セッション内の感情弧
    arc := sg.currentArc(now)
    switch arc {
    case ArcBreakthrough:
        parts = append(parts, "さっきなんかうまくいったっぽい気がしました")
    case ArcGrind:
        parts = append(parts, "ずっと根気よく続けてます")
    case ArcMomentum:
        parts = append(parts, "最近ペースが上がってる感じがします")
    }

    // failStreak（行き詰まりシグナル）
    if sg.failStreak >= 2 {
        parts = append(parts, "少し手こずってる感じがしてます")
    }
}
```

### 案E: ユーザーリアクション学習（インパクト大・コスト大）

**現状:** どのセリフがユーザーに「響いた」か記録していない。

**提案:** UserClickのタイミングをリアクション信号として使う。

```
仮説: セリフが表示されてから30秒以内にUserClickが来た = そのセリフに反応した

記録:
- speech → reaction: positive
- speech → no reaction（次の speech まで無視）: neutral/negative

活用:
- 高評価だったセリフのカテゴリ・長さ・語調パターンを学習
- バッチ生成のexamplesをそのユーザーの好みで動的に差し替え
```

### 案F: プライベートトーク（インパクト大・コスト中）

**動機:**
作業コメントだけだとサクラが「応援ツール」になってしまい、「隣にいる子」感が薄い。
女の子がそばで働きながら、ふと自分のことを話す・何気なく話しかけてくる瞬間が、
実は一番モチベーションになる（同伴感・特別感）。

**コンセプト: 「ねえ、急に聞いていいですか」**

作業と無関係に、サクラ自身の小さな感情・関心・疑問を差し挟む発言枠を作る。
ユーザーを褒めたり観察したりするのではなく、**サクラが主語になって喋る**。

```
例:
「ねえ先輩、急に聞いていいですか」（前置きだけで終わる・次のクリックで続く）
「私さ、なんか今日すごく眠くて、集中できてない感じがします」
「なんか最近、夜遅いと変な夢みる気がして、先輩もそういうことないですか」
「私って、雨の日のほうがなんか集中できる気がするんですよね」
「さっきなんか急においしいもの食べたくなって、全然関係ないんですけど」
「先輩って休憩するとき何してます？私なんか全然決まってなくて」
「なんか最近ずっとここにいるから、私もちょっと慣れてきた感じがします」
「私、こういう静かな時間って嫌いじゃないんですよね」
```

**特徴:**
- 作業内容・技術・観察を一切含まない → 的外れになりようがない
- サクラが「キャラクターとして存在している」感が出る
- ユーザーが「返したくなる」自然な会話トリガーになる（UserQuestion誘導）
- 汎用励ましフレーズの禁止リストが不要（そもそも励まし文ではない）

**トリガー設計:**

idle/heartbeat タイミングのうち、一定確率（20〜30%）でこのカテゴリを使う。

```go
// pool category 選択時
if category == "heartbeat" && rand.Float32() < 0.25 {
    category = "private" // プライベートトークに差し替え
}
```

プールキー: `"cute:private:ja"`, `"genki:private:ja"` など

**ja.yaml examples設計:**

```yaml
examples:
  private:
    working:  # 作業中の「ふと話しかける」系
      - "私さ、急に聞いていいですか"
      - "なんか今日、なんとなく眠くて、全然関係ないですけど"
      - "ねえ、先輩って休憩するとき何してます？"
      - "私ってなんか、雨の日のほうが落ち着く気がするんですよね"
      - "さっきふと、おいしいもの食べたくなっちゃって"
      - "こういう静かな時間、私は嫌いじゃないです"
      - "先輩と一緒にいるの、なんか慣れてきた感じがします"
      - "なんか最近ちょっと、いろいろ気になってることがあって"
      - "私なんか、夜になると急にいろんなこと考えちゃうんですよね"
      - "ねえ、なんかいいことありましたか、急に聞いてみたくなって"
```

**性格別の差異:**
- `genki`: 「ねえねえ！急に聞いていいですか？」エネルギー高め
- `cute`: 「あの…急に変なこと聞くんですけど」恥ずかしそう
- `oneesan`: 「一個聞いていいですか」落ち着いたトーン
- `cool`: 「ちょっと関係ない話なんですけど」ぶっきらぼうに

**注意点:**
- 頻度を出しすぎると「話しかけてくる迷惑な子」になる → 4〜5セリフに1回程度
- 同じ話題が続かないよう AlreadyShown 重複チェックは他カテゴリより厳しくする
- プライベートトークの後にUserClickがあった場合 → UserQuestion誘導で会話が続くと最高

## 5. 現状の whack-a-mole から抜け出すための根本対策

### 問題の本質

LLMは「何かポジティブなことを言え」という曖昧な指示に対し、
訓練データの中の「汎用的な励まし語彙」（絶対いけます・いい感じ・素敵ですね）に収束する。

### 根本対策

**プロンプトの「禁止ルール」より「役割ロールプレイの精度」を上げる。**

現状の `[プロセス]` セクション（ja.tmpl 141行目〜）は良い方向だが、
さらに「サクラが感じていること」の具体的な出発点を毎回与える。

```
改善アイデア:
[今サクラが感じていること]
{{.EmotionalContext}}
（例: "さっきから30分、ずっと向き合ってるのを見て、こっちもなんか落ち着かなくて"）

→ この1行があることで、LLMが汎用フレーズではなく
  「この感情から始まる言葉」を生成するようになる
```

`EmotionalContext` は Go 側で `currentArc()` と `SituationHint` から組み立てる。
これはプロンプト工学的に「感情の入口を与える」アプローチで、
禁止リストより根本的に品質を上げる可能性が高い。

## 6. まとめ

| 現状 | 目指す姿 |
|------|---------|
| 観察コメント（的外れになりやすい） | 感情的同伴（ユーザーの状態に共鳴する） |
| whack-a-mole（悪いフレーズを禁止） | ポジティブ制約（感情の入口を与える） |
| 均一な working カテゴリ | arc に応じた感情フェーズ別セリフ |
| 生成時文脈ほぼなし | SituationHint + EmotionalContext を常に注入 |
| 発言頻度は時間ベース | 感情的タイミングで優先発言 |

**最短で効果が出る施策:** 案C（タイミング最適化）+ 案D（SituationHint強化）
**中長期で品質を根本改善する施策:** 案A（セッションアーク）+ EmotionalContext注入
